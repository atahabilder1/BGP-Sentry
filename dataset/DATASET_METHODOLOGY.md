# BGPSentry Dataset — Generation Methodology (v3, 2026-04)

## 1. Purpose

This dataset supports the **BGPSentry** paper evaluating a blockchain-based
BGP monitoring system with Proof-of-Population consensus. The primary
metrics it enables are **scalability** (network TPS, per-node TPS, consensus
latency, P2P message growth, fork resolution rate) across five network
sizes. Secondary metrics include per-attack-type detection coverage.

The dataset is **CAIDA-anchored simulation**, not trace-driven. Topology,
AS relationships, and RPKI cryptographic identity are derived from real
measurement sources; BGP announcements, per-AS observations, and attack
scenarios are generated by BGPy's Gao-Rexford simulation engine over the
real topology.

## 3. Data Provenance — What Is Real vs. Synthetic

| Component | Source | Type |
|---|---|---|
| AS identities (real ASNs) | CAIDA 2022-01-01 | **Real** |
| Customer-provider links | CAIDA (inferred from BGP feeds) | **Real** |
| Peer links | CAIDA (inferred from BGP feeds) | **Real** |
| AS tier classification (clique/transit/mh/stub) | CAIDA-derived via BGPy `asn_groups` | **Real** |
| RPKI signing status | rpki-client 2022-06-18 VRPs (NLnet Labs / OpenBSD) | **Real (with normalization, see §5)** |
| Per-AS prefixes | pyasn (RouteViews RIB) when available | **Real** |
| Gao-Rexford propagation | BGPy SimulationEngine | **Standard model** |
| Announcement timing | Weibull inter-arrival (shape=0.7) | **Empirically modeled** |
| BGP attributes (LOCAL_PREF, MED, ORIGIN) | Gao-Rexford + RFC 4271 | **Standard model** |
| Per-hop convergence jitter | MRAI 1–30s per hop | **Standard model** |
| Attack scenarios | BGPy custom scenarios | **Synthetic (real categories)** |
| Attacker/victim selection | Random from sampled ASes | **Synthetic** |
| Attack rate | Elevated ~0.3–1% for statistical evaluability | **Synthetic** |

The dataset is **topologically real** and **crypto-identity real** but
**traffic-synthetic**. This is standard practice for BGP security
simulation work (Furuness et al. NDSS 2025, Saad et al. Comp. Networks
2022) — nobody uses synthetic trace datasets claiming real-world
volumes. See §6 for what the paper should and should not claim.

## 4. Topology Sampling (v3)

### 4.1 Connected subgraph extraction

The generator samples a connected subgraph of the real CAIDA 2022-01-01
AS-level topology via tier-stratified selection with bridge reconnection.

**Seed quotas** (per `extract_caida_subgraph` in `generate_rpki_dataset.py`):
- **Clique floor**: `max(1, min(5, int(N * 0.04)))` — at most 5 tier-1 ASes.
  Unlike the previous version which forced all 19 input_clique ASes (a
  major cause of the old 43–69% RPKI inflation), this keeps backbone
  representation proportional.
- **Transit**: `(N − clique) × 0.18`
- **Multihomed**: `(remaining) × 0.50`
- **Stubs**: `remaining`

**Bridge reconnection**: stub-dominated seed samples are heavily
fragmented (dry-run showed ~90% of sampled ASes as singleton components
at N=50). To restore connectivity, the generator walks BFS paths in the
full CAIDA graph from each disconnected component to the main component
and adds all real intermediate ASes. Bridging is **unlimited** —
connectivity is a correctness requirement, not a size budget.

**Consequence**: actual output size exceeds the seed quota by ~40–100%.
A `caida_100` dataset has ~104 ASes, a `caida_1250` dataset has ~1305 ASes.
Nominal names reflect the intended scale; actual sizes are reported in
each dataset's `as_classification.json`.

### 4.2 Typed relationship preservation

Edges in the sampled subgraph are **real CAIDA relationships**. The
generator iterates each sampled AS's `providers`, `customers`, and `peers`
lists separately and constructs typed `CPLink` and `PeerLink` objects for
edges where both endpoints are in the sample. No synthetic edges are
introduced; Gao-Rexford valley-free routing over the subgraph uses real
customer-provider-peer semantics.

### 4.3 Known structural biases

Connected subgraphs of CAIDA are **backbone-biased** because bridge
reconnection walks through transit hubs. Compared to the global CAIDA
average:

- **Lower average degree** — sampling drops edges where only one endpoint
  is in the sample
- **Over-representation of transit tier** — bridging pulls in high-degree
  core ASes
- **Under-represented peering mesh** — most IXP peering is lost

This is a fundamental property of connectivity-preserving subsampling,
not a bug. The paper's scalability claims are robust to this bias because
the metrics (network TPS, consensus latency, P2P overhead) depend on
blockchain mechanics and number of validators, not on raw degree
distribution.

## 5. RPKI Classification

### 5.1 Source: rpki-client VRPs, not rov-collector

This dataset defines "RPKI-enabled" as **AS with at least one registered
ROA** (the RPKI signing property), **not** as "AS that enforces ROV" (the
stricter enforcement property). For BGP-Sentry's Proof-of-Population
consensus, signing is the correct metric because what anchors a validator
is cryptographic identity rooted at a RIR Trust Anchor Location, not
inbound filtering policy.

**Source**: NLnet Labs / OpenBSD `rpki-client` Validated ROA Payload (VRP)
dump from https://console.rpki-client.org/vrps.json. The canonical output
of the canonical RPKI validator, covering all 5 RIR TALs (ARIN, APNIC,
RIPE NCC, LACNIC, AFRINIC).

**Snapshot**: 2022-06-18 via Wayback Machine
(https://web.archive.org/web/20220618163743/https://console.rpki-client.org/vrps.json)
— temporally aligned with the CAIDA 2022-01-01 topology snapshot.

**Coverage**: 30,849 ASes globally have at least one ROA. Of these, 26,631
are present in the CAIDA 2022-01-01 graph, yielding a **global AS-level
RPKI signing rate of 36.3%**.

### 5.2 Label normalization

Connected CAIDA subgraphs are **structurally biased above** the global
36.3% rate because bridge reconnection walks through backbone transit
ASes, which have ~65% RPKI signing rate (vs. 36% global). The natural
sample rate is 47–55% (dry-run verified).

To make the dataset reflect realistic global deployment density rather
than backbone bias, the generator **uniformly downsamples RPKI labels**
in each subgraph until the final ratio matches the global 36.3%:

1. Compute the natural RPKI set from rpki-client VRPs (real data).
2. If natural count > `round(N * 0.363)`, randomly demote the excess ASes
   to non-RPKI using a dedicated seeded RNG (`random.Random(("label_norm",
   seed, N))`) separate from the sampling RNG.
3. Demoted ASes retain their real rpki-client status in metadata but are
   marked non-RPKI in `as_classification.json` for simulation purposes.

**Per-dataset normalization counts**:

| Dataset | Natural RPKI | Target | Demoted | Final |
|---|---|---|---|---|
| `caida_100`  | 58 / 104  | 38 | 20 | 38 / 104  (36.54%) |
| `caida_200`  | 100 / 210 | 76 | 24 | 76 / 210  (36.19%) |
| `caida_350`  | 184 / 347 | 126| 58 | 126 / 347 (36.31%) |
| `caida_650`  | 333 / 645 | 234| 99 | 234 / 645 (36.28%) |
| `caida_1250` | 638 / 1305| 474|164 | 474 / 1305 (36.32%) |

Label normalization is a **controlled experimental adjustment**,
transparently documented, deterministic, and reproducible. It is standard
practice in simulation sensitivity studies.

## 6. Simulation Duration: 600 seconds (10 min)

All datasets use a **600-second (10-minute) wall-clock simulation window**,
matching the 300-second trust-evolution figure in the paper with a
conservative 2× margin for scalability-metric stability.

Justification:
- **BGPy's native mode is round-based**, not time-based — no wall-clock
  window exists in its default operation. The 10-minute window is layered
  on via `RealisticTimeline` (Weibull-sampled inter-arrival times).
- The paper's Figure 4 (trust score evolution) already uses 300 seconds.
- Related blockchain-BGP work (RouteChain) reports per-transaction latency
  in **milliseconds** and does not use a wall-clock window at all.
- At 3.2s consensus latency, 600s yields ~187 consensus rounds per
  validator — more than sufficient for stable TPS, latency, and fork
  statistics.
- 10 min × 600 ASes × Weibull bursty traffic produces 50K–250K announcements
  per dataset, enough to saturate per-node TPS measurement.

## 7. Attack Scenarios

Six attack types, all from BGPy's `custom_scenarios`:

| Type | Pre/Post-ROV | BGPy class |
|---|---|---|
| PREFIX_HIJACK | Pre-ROV | `PrefixHijack` |
| SUBPREFIX_HIJACK | Pre-ROV | `SubprefixHijack` |
| BOGON_INJECTION | Pre-ROV | `BogonInjection` |
| ROUTE_FLAPPING | Pre-ROV | `RouteFlapping` |
| FORGED_ORIGIN_PREFIX_HIJACK | **Post-ROV** | `ForgedOriginPrefixHijack` |
| ACCIDENTAL_ROUTE_LEAK | Pre/Post-ROV | `AccidentalRouteLeak` |

The inclusion of post-ROV attacks (forged-origin) and route leaks
validates BGP-Sentry's core claim: that it extends BGP security beyond
what RPKI alone can detect.

**Attack count scaling**: `attacks-per-type` varies by size (3 for
N=100/200, 4 for N=350, 7 for N=650, 13 for N=1250). Per-dataset attack
counts are in each `ground_truth/ground_truth.json`.

**Attack rate**: Elevated above real-world frequencies (~0.01%) to
~0.3–1% of total announcements for statistical evaluability. This is
standard practice in simulation-based BGP security research and should be
documented in the paper's methodology section.

## 8. Per-AS Observations

Each AS in each dataset has its own `observations/AS<asn>.json` file with
the structure required by the BGP-Sentry loader at
`nodes/rpki_nodes/shared_blockchain_stack/data_loader.py`:

```json
{
  "asn": 12345,
  "is_rpki_node": true,
  "total_observations": 5173,
  "best_route_observations": 4890,
  "alternative_route_observations": 283,
  "attack_observations": 12,
  "legitimate_observations": 5161,
  "observations": [
    {
      "prefix": "44.12.0.0/16",
      "origin_asn": 6789,
      "timestamp": 1775816438,
      "is_attack": false,
      "label": "LEGITIMATE",
      "as_path": [12345, 6789],
      ...
    }
  ]
}
```

Required per-observation fields (enforced by the loader): `prefix`,
`origin_asn`, `timestamp` (Unix epoch > 1e9), `is_attack`, `label`,
`as_path`. Additional fields (`as_path_length`, `recv_relationship`,
`origin_type`, `bgp_update`, `communities`, `is_best`, `hop_distance`,
`timestamp_readable`, `is_withdrawal`) are included for analysis but are
not read by the loader.

## 9. Ground Truth

- `ground_truth/ground_truth.json` — attack-labeled announcements in the
  same schema as observations, used by post-hoc analysis
- `ground_truth/ground_truth.csv` — compact CSV view (observer, timestamp,
  attack_type, attacker, prefix) for easy grep/inspection
- `ground_truth/as_classification.json` — duplicate of the root
  `as_classification.json` for backward compatibility

## 10. What the Paper Can Honestly Claim

### Can claim ✅
- *"Evaluated on real CAIDA AS-level topology with real customer-provider
  and peer relationships."*
- *"RPKI status derived from NLnet Labs/OpenBSD rpki-client Validated ROA
  Payloads (2022-06 snapshot), normalized to the global 36.3% AS-level
  signing rate."*
- *"Six real-world attack categories including post-ROV forged-origin
  hijacks and route leaks, validating BGP-Sentry's coverage beyond
  conventional ROV."*
- *"Scalability curves across five dataset sizes (100, 200, 350, 650,
  1250 ASes), measuring network TPS, consensus latency, fork resolution,
  and P2P message growth."*

### Cannot honestly claim ❌
- ~~"Trace-driven"~~ — dataset is CAIDA-anchored simulation, not replay of
  real BGP feeds.
- ~~"Representative of real Internet traffic volumes"~~ — announcement
  rate is synthesized by BGPy.
- ~~"Real-world attack detection rates"~~ — attack rate is elevated for
  statistical power.
- ~~"Full Internet scale"~~ — 1250 max vs. 73K real CAIDA ASes.

## 11. Reproducibility

Every dataset includes a `as_classification.json` with:
- `rpki_source_url` — exact Wayback Machine URL
- `rpki_natural_count` and `rpki_demoted_count`
- `rpki_target_ratio` (0.363)
- `topology_source` and `subgraph_method`
- `visibility_stats`

To reproduce: check out `bgpy_pkg/generate_rpki_dataset.py` at the same
git commit, run with the same `--seed` and `--nodes` values, and the
results will be bitwise identical (modulo timestamp jitter which is also
seed-derived).
